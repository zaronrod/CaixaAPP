Visão do Produto

O CaixaAPP é um aplicativo Android de controle financeiro pessoal e familiar.
Ele permite registrar receitas e despesas, classificar lançamentos por pessoa ou família, 
aplicar rateios automáticos definidos por regras externas e visualizar os dados tanto em 
formato de extrato quanto em gráficos comparativos ao longo do tempo.

O foco do produto é clareza financeira, rastreabilidade dos lançamentos e coerência matemática entre telas.

ÉPICO 1

Autenticação e Acesso Seguro

Objetivo

Garantir que apenas o dono do dispositivo tenha acesso aos dados financeiros, usando mecanismos nativos do Android.

Histórias de Usuário

Como usuário
Quero acessar o aplicativo usando a autenticação do meu próprio celular
Para garantir que meus dados financeiros não sejam acessados por terceiros

Critérios de aceite
O app deve solicitar autenticação biométrica ao iniciar
Caso o dispositivo não tenha biometria, deve usar PIN ou senha do sistema
Após autenticação bem sucedida, o usuário é direcionado para a tela de lançamento
Falhas de autenticação não devem permitir acesso parcial ao app

ÉPICO 2

Cadastro de Lançamentos Financeiros

Objetivo

Permitir que o usuário registre entradas e saídas financeiras de forma simples, consistente e validada.

Histórias de Usuário

Como usuário
Quero registrar um lançamento financeiro com valor, descrição, data, tipo e pessoa
Para manter controle detalhado das minhas finanças

Critérios de aceite
Valor deve ser numérico e maior que zero
Descrição não pode estar vazia
Data deve ser obrigatória e selecionável via calendário
Tipo deve permitir escolher crédito ou débito
Pessoa deve ser selecionada a partir de uma lista carregada de arquivo JSON local
Ao salvar, o lançamento deve ser persistido no banco configurado

ÉPICO 3

Gestão de Pessoas e Rateio Familiar

Objetivo

Permitir classificação de lançamentos por pessoa e cálculo automático de rateios familiares.

Histórias de Usuário

Como usuário
Quero classificar lançamentos como familiares ou pessoais
Para que o sistema calcule automaticamente minha parte nos gastos e ganhos da família

Critérios de aceite
A lista de pessoas deve ser carregada de um JSON local
As regras de rateio devem ser lidas de um segundo JSON
Lançamentos da família não pertencem diretamente a indivíduos
Ao visualizar extratos individuais, o sistema deve aplicar o rateio correto
O rateio deve funcionar tanto para créditos quanto para débitos
O rateio não deve ser salvo no banco, apenas calculado em tempo de consulta

ÉPICO 4

Extrato Financeiro

Objetivo

Exibir os lançamentos financeiros de forma organizada, filtrável e matematicamente consistente.

Histórias de Usuário

Como usuário
Quero visualizar um extrato financeiro filtrado por pessoa ou família
Para entender meus gastos, ganhos e saldo final

Critérios de aceite
O extrato deve permitir seleção de pessoa ou família
Quando filtrado por pessoa, deve incluir lançamentos pessoais e rateios familiares
Quando filtrado por família, deve exibir apenas lançamentos familiares com valor integral
Os lançamentos devem exibir descrição, data, tipo e valor
A lista deve ser ordenada por data decrescente no formato DD/MM/YYYY
Créditos devem ser visualmente diferenciados de débitos
O saldo final deve ser exibido com base nos valores apresentados

ÉPICO 5

Gráficos de Análise Temporal

Objetivo

Fornecer uma visão analítica do fluxo financeiro ao longo do tempo.

Histórias de Usuário

Como usuário
Quero visualizar um gráfico comparando créditos e débitos dos últimos seis meses
Para entender tendências e resultados financeiros recentes

Critérios de aceite
O gráfico deve considerar exatamente os últimos seis meses
Os dados devem respeitar as mesmas regras de filtro e rateio do extrato
Cada mês deve exibir o total de créditos e débitos
O eixo horizontal deve representar os meses
O eixo vertical deve representar valores monetários
Ao final do gráfico deve ser exibido o resultado líquido dos seis meses
O cálculo do resultado líquido deve seguir a mesma lógica do extrato

ÉPICO 6

Persistência e Arquitetura

Objetivo

Garantir organização de código, facilidade de manutenção e independência de banco.

Histórias de Usuário

Como desenvolvedor
Quero uma arquitetura clara separando modelo, controle e visualização
Para facilitar manutenção, testes e evolução do app

Critérios de aceite
O projeto deve seguir MVC
A persistência deve ser acessada via interface de repositório
Deve ser possível alternar entre Room e Firebase sem alterar a UI
Nenhuma lógica de negócio deve ficar na camada de visualização
